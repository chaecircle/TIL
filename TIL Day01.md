## Github 깃허브 특강



### Why Git and Github?

1. 깃Git 과 깃허브Github 란 무엇인가?

 깃Git은 분산 버전 관리 프로그램이다. 어떠한 작업을 할 때, 원본과 수정사항을 반영한 수정본들을 체계적으로 관리해줌으로써 사용자에게 편의를 제공한다. 

 여기서 분산이 가리키는 말은 그 반댓말을 먼저 알면 쉽다. 중앙 집중식이 그 반댓말인데, 파일의 원본과 그 변경사항들을 특정 중앙에서 관리하는 것이 바로 그것이다. 분산은 이와 반대로 모든 사용자들에게 분산되어 각각 원본과 변경사항들이 저장되는 방식인데, 블록체인과 유사하게 어느 한곳에 문제가 생기더라도 분산되어 저장되므로 안정성이 뛰어나다는 장점이 있다. 

 깃허브는 마치 개발자들의 인스타그램과도 같다. 자신의 개발 경험과 행적들을 모두 기록하고 담을 수 있는 공간이다. 따라서 포트폴리오를 쌓기에 아주 좋은 플랫폼이라고 할 수 있다. 



## CLI이란

> Command Line Interface의 약자

 오늘날의 퍼스널 컴퓨터(PC)를 탄생시킨 GUI의 이전 컴퓨터들을 작동했던 기본적인 인터페이스 방식으로, 한 줄 한 줄 명령어로 컴퓨터를 다루는 방식이다.



> UNIX 운영 체제에서 개발된 프로그램이 많기에, 기존에 존재하는 윈도우의 powershell이나 명령 크롬프트가 아닌, UNIX 운영 체제로 번역해주는 Git Bash를 사용하여 CLI를 익혀보자. 



##### 1. 루트, 홈 디렉토리

< 루트 디렉토리 >

- 모든 파일/폴더를 담고 있는 폴더. 가장 상위의 폴더를 말한다. 
- windows 같은 경우, 보통 C 드라이브를 의미한다. 

< 홈 디렉토리 >

- Tilde틸드 라고도 칭하며, 현재 로그인 된 사용자의 홈 폴더를 의미한다.



##### 2. 절대 경로와 상대 경로

< 절대 경로 >

- 루트 디렉토리부터 목적 지점까지 거치는 모든 경로를 전부 작성한 것



< 상대 경로 >

- 현재 워킹 디렉토리를 기준으로 계산된 경로를 말함. 
- 절대 경로의 반대어.



상대 경로로 쓰이는 명령어들의 예시는 다음과 같다. 

(1) . : 현재 워킹 디렉토리

(2) .. : 현재 워킹 디렉토리의 바로 상위 폴더



##### 3. 터미널 명령어

1. `touch`

- 파일을 생성하는 명령어

- 띄어쓰기로 구분하여 여러 파일을 한 번에 생성 가능

  `touch 일기장.txt 과제1.txt 과제2.txt`

- 숨김 파일을 만들기 위해서는 파일 앞에 `.`을 붙이면 된다.



2. `mkdir`

- make directory 의 약자로, 새 폴더를 생성하는 명령어

- 띄어쓰기로 구분하여 여러 폴더 생성 가능

- 폴더 이름 사이에 공백이 있다면, 따옴표로 이름을 묶어준다

  ```bash
  mkdir 이화여자대학교
  mkdir '제37대 심리학과 학생회 SYM;PHONY'
  ```



3. `ls`

- list segment의 약자로, 현재 작업 중인 디렉토리의 폴더/파일 목록을 보여주는 명령어.
- `-a` : all 옵션. 숨김 파일까지 모두 보여준다. 
- `-l` : long 옵션. 용량, 수정 날짜 등 파일 정보를 자세히 보여준다. 
- `-l -a`/`-al`: 이처럼 여러가지 옵션을 중첩하여 함께 사용할 수 있다. 



4. `mv`

- move의 약자로, 폴더/파일을 다른 폴더 내로 이동하거나 이름을 변경하는 명령어. 

- 사용 방법: `mv (파일명) (폴더명)`

- 단, 다른 폴더로 이동할 때는 작성한 폴더가 반드시 있어야 한다. 없다면 **이름이 바뀐다.** 

  ```bash
  #Saint.txt를 S_class 폴더 안에 넣을 때
  mv Saint.txt S_class
  
  #Saint.txt의 이름을 Honey.txt로 바꿀 때
  mv Saint.txt Honey.txt
  ```



5. `cd`

- change directory의 약자로, 현재 작업중인 디렉토리를 변경하는 명령어

- 사용 방법: `cd (이동할 디렉토리명)`

  ```bash
  #현재 작업중인 디렉토리에 있는 Reader 폴더로 이동
  cd Reader
  
  #절대 경로를 통한 디렉토리 변경
  cd C:/Users/scw/Desktop
  
  #상대 경로를 통한 디렉토리 변경
  	#부모 디렉토리로 가기(위로 가기)
  	cd ..
  	#바로 전 디렉토리로 가기(뒤로 가기)
  	cd -
  	#홈 디렉토리로 이동(~을 쓰지 않아도 무방)
  	cd ~ 
  ```



6. `rm`

- remove의 약자로, 폴더/파일을 지우는 명령어

- GUI에서의 삭제와 달리 휴지통으로 이동하지 않는 __영구 삭제__이므로 주의해서 사용해야 한다. 

- 사용 방법: `rm (폴더/파일명)`

- `*(asterisk)`를 사용하여 `rm *.txt`라고 입력하면, txt 파일 전체가 삭제된다. 

- `-r` : recursive 옵션으로 폴더를 지울 때 사용한다. 

  ```bash
  rm Saint.txt
  rm -r S_class
  ```

  

7. `start`

- 폴더/파일을 여는 명령어
- Windows에서는 start를 사용하지만, Mac에서는 open을 사용한다.
- 사용 방법: `start (파일명/폴더명)`



8. __유용한 단축키__

- __위, 아래 방향키__: 과거에 작성했던 __명령어 조회__
- `tab`: 폴더/파일 이름 자동 완성
- `ctrl + a`: 커서가 맨 앞으로 이동
- `ctrl + e`: 커서가 맨 뒤로 이동
- `ctrl + w`: 커서가 앞 단어를 삭제
- __`ctrl + l`__: 터미널 화면을 깨끗하게 __청소__. (스크롤 올리면 과거 내역 조회 가능)
- `shift + insert`: 붙여넣기

## Markdown



#### (1) 마크다운이란?

- 일반 텍스트 기반의 경량 마크업Markup 언어

  > 마크업Markup이란?
  >
  > - 마크Mark로 둘러싸인 언어로, 마크는 글의 역할을 지정하는 일종의 표시이다. 
  > - 예를 들어, HTML에서 M은 Markup의 약자이다. 

- 마크업을 더 쉽고, 간단하게 사용하고자 만들어졌다. 





#### (2) 마크다운의 장점과 단점 및 주의사항



1. __장점__

   - 문법이 직관적이며, 쉽다.
   - 관리가 용이하다.
   - 지원 가능한 플랫폼과 프로그램이 다양하다. 

2. __단점__

   - 표준이 없어 사용자마다 문법이 상이할 수 있다. 
   - 모든 HTML의 마크업 기능을 대신할 수 없다. 

3. __주의사항__

   - 마크다운의 본질은 글에 __역할을 부여__하는 것이다.

   - 따라서, 반드시 __역할__에 맞는 마크다운 문법으로 작성해야 한다. 

     ** 예를 들어, 글씨의 크기를 키우고 싶다는 이유로 '내용'에게 '제목'의 역할을 부여해서는 안 된다.





#### (3) 마크다운 문법



1. __제목(Headings)__

- `h1 ~ h6` 에 해당하는 제목을 표현

- 사용 방법: `#`을 사용한다. 

  ```bash
  # 제목 1 (가장 글씨 크기가 큼)
  ## 제목 2
  ### 제목 3
  #### 제목 4
  ##### 제목 5
  ###### 제목 6 (가장 글씨 크기가 작음)
  ```

  

2. **목록(List)**

- 순서가 있는/없는 목록

- 순서 없는 목록은 `- * +`를 사용한다. 

- 순서 있는 목록은 `1. 2. 3. ` 처럼 숫자를 사용한다. 

- `tab 키` 를 이용하여 들여쓰기를 할 수 있다. 

  ```bash
  - 순서 없는 목록
  	- 서브 목록
  	- 서브 목록
  	
  + 순서 없는 목록
  	+ 서브 목록
  	+ 서브 목록 
  		+ 서서브 목록
  		
  1. 순서 있는 목록
  	- 혼합 서브 
  	- 혼합 서브
  ```

  

3. __강조(Emphasis)__

- 글자의 스타일링을 표현한다. 
  - _기울임_ : `*글자*` 혹은 `_글자_` 
  - __굵게__ : `__글자__` 혹은 `**글자**`
  - ~~취소선~~ : `~~글자~~` 



4. __코드(Code)__

- 한 줄 코드인 __인라인 코드__와 여러 줄 코드인 __블록 코드__로 나뉜다. 
  - 인라인 코드 Inline Code : `(백틱)inline code(백틱)`
  - 블록 코드 Block Code : ````python` 처럼 백틱을 3번 입력하고, 코드의 종류를 작성. (eg. python, Bash, markdown, javascript) 



5. __링크(Links)__

- 클릭하면 해당 주소로 이동할 수 있는 링크를 표현할 수 있다. 

- `[표시할 글자] (이동할 주소)` 형태로 작성한다. 

  ```markdown
  [google](http://google.com)을 눌러서 구글로 이동하세요
  ```

  - 표현 : [google](http://google.com)을 눌러서 구글로 이동하세요 

  

6. __이미지(Images)__

- `! [대체 텍스트](이미지 주소)` 형태로 작성한다. Typora에서는 이미지 파일을 끌어와서 높아도 자동 업로드 된다.  
- [대체 텍스트]란, 이미지를 정상적으로 불러오지 못했을 때 표시되는 문구이다. 



7. __인용(Blockquote)__

- `>`를 사용하며, 갯수에 따라 중첩 사용이 가능하다. 

  > 인용문1
  >
  > > 중첩 인용1
  > >
  > > > 중첩 인용2



8. __표(Table)__

- 파이프(|)와 하이픈(-)을 이용하여 행과 열을 구분한다. 
- 테이블 양쪽 끝의 파이프(|)는 생략 가능하다. 
- 헤더 셀을 구분할 때는 3개 이상의 하이픈(-)이 필요하다. 
- Typora에서는 `ctrl + T`로 쉽게 표 생성이 가능하다. 
  - 행을 늘릴 때는 `ctrl + enter` 사용. 



9. __수평선(Horizontal Rule)__

- `- 또는 * 또는 _`을 3번 이상 연속으로 작성한다. 





## Git 기초



#### (1) Git 초기 설정

> 최초 한 번만 설정하는 것이다. 



1. 작성자를 확인할 수 있도록 이름과 이메일을 설정한다. 

```bash
git config --global user.name "한유진"
git confug --global user.email "Saint0830@~"
```

- 작성자를 수정할 때에는 이름, 메일 주소만 다르게 하여 동일하게 재입력한다. 

2. 작성자가 올바르게 설정되었는지 확인 가능하다.

```bash
git config --global -l 
또는
git config --global --list
```





#### (2) Git 기본 명령어

> Git에는 세 가지 종류의 공간이 존재한다. 분장실(대기실)/무대/촬영 으로 비유할 수 있다. 

- Working Directory(=Working Tree) : 사용자의 일반적인 작업이 일어나는 곳
- Staging Area(=Index) : 커밋을 위한 파일 및 폴더가 추가되는 곳
- Repository : staging area에 있던 파일 및 폴더의 변경사항(커밋)을 저장하는 곳
- Git은 working directory -> staging area -> repository 의 과정으로 버전 관리를 수행한다. 
  - 이렇게 공간이 세 가지로 분리되어 있기 때문에 사용자는 수정사항을 원하는 대로 더 편리하고 유연하게 업데이트 및 관리할 수 있다. 



1. __`git init`__

- 현재 작업 중인 디렉토리를 Git으로 관리한다는 명령어.
- .git이라는 숨김 폴더를 생성하고, 터미널에는 `(master)`라고 표기된다. 
  - 차별적인 뉘앙스가 포함된 master가 아닌, main이라는 용어로 대신되는 추세지만, 현재까지는 master가 지배적인 위치에 있다. 
  - 주의사항: 절대로 __홈 디렉토리에서 git init을 하지 않도록__ 주의한다!



2. __`git status`__

- working directory(이하 w.d.)와 staging area(이하 s.a.)에 있는 파일의 현재 상태를 알려주는 명령어. 

- 작업을 시행하기 전, 수시로 확인하는 습관을 들이자. 

- 상태:

  - `Untracked` : Git이 관리하지 않는, 한 번도 s.a.에 올라간 적 없는 파일. 

  - `Tracked` : Git이 관리하는 파일

    ​	a. `Unmodified` : 최신 상태

    ​	b. `Modified` : 수정되었지만, 아직 s.a.에 반영하지 않은 상태

    ​	c. `Staged` : s.a.에 올라간 상태

​		

3. __`git add`__

- w.d.에 있는 파일을 s.a.로 올리는 명령어
- Git이 해당 파일을 추적(관리)할 수 있도록 만든다.



4. __`git commit`__

- s.a.에 올라온 파일의 변경 사항을 하나의 버전(커밋)으로 저장하는 명령어
- 커밋 메시지는 현재 변경 사항들을 잘 나타낼 수 있도록 의미있게 작성하는 것이 좋다. 
  - 참고: [ 좋은 커밋을 쓰기 위한 7가지 약속](https://meetup.toast.com/posts/106)
- 각각의 커밋은 SHA-1 알고리즘에 의해 반환된 고유의 해시값을 ID로 가진다. 
- `(root commit)`은 해당 커밋이 최초의 커밋일 때만 표시된다. 



5. __`git log`__

- 커밋의 내역(Id, 작성자, 시간, 메시지 등)을 조회할 수 있는 명령어
- 옵션
  - `--oneline` : 한 줄로 축약해서 보여준다.
  - `--graph` : 브랜치와 머지 내역을 그래프로 보여준다.
  - `--all` : 현재 브랜치를 포함한 모든 브랜치의 내역을 보여준다.
  - `--reverse` : 커밋 내역의 순서를 반대로 보여준다. (최신은 가장 아래)
  - `-p` : 파일의 변경 내용도 같이 보여준다.
  - `-(숫자)` : 원하는 갯수만큼의 내역을 보여준다. (eg. `-3`)



##### ** 참고: 옵션과 인자 **

- 옵션
  - 옵션은 명령어의 __동작 방식__을 지정한다. 따라서 __생략 가능__하다. 
  - 부가적인 기능을 원할 때 사용한다. 
  - eg. `git log --oneline` 와 같이 `--oneline`옵션은 부가적인 기능이므로 생략해도 `git log`는 정상 동작한다. 

- 인자
  - 인자는 명령어의 __동작 대상__을 지정한다. 따라서 __생략이 불가__하다. 
  - eg. `git add`라고만 작성하면 어떤 대상을 s.a.에 올릴지 알 수가 없으므로 동작이 이뤄질 수 없다. 따라서 `git add healer.txt`와 같이 `healer.txt`라는 인자를 추가해주어야 한다. 





## Github

> [Github 주소](https://github.com/)로 접속하여 회원가입을 한다. 
>
> 개발자로서의 이름이 될 닉네임은 신중하게 작성하는 것이 좋다. 
>
> 사용자 이메일 주소는 `git config`설정에서 작성했던 이메일과 동일하게 기입하여야 한다. 





#### (1) 원격 저장소 Remote Repository

1. Github에서 Repository를 추가하여 생성할 수 있다. 
2. 로컬 저장소(내 컴퓨터에 생성되어 있는 저장소)와의 연결해야 하는 repositotry의 주소를 복사하여 연결한다. 방법은 다음과 같다. 

- 원격 저장소 등록 : `git remote add <주소 명명> <주소url>`  
  - eg. `git remote add origin http://github.com/chaecircle/TIL.git`
- 원격 저장소 조회: `git remote -v`
- 원격 저장소 연결 삭제: `git remote rm <명명된 주소 이름>` 혹은 `git remote remove <명명된 주소 이름>`
  - _연결을 끊는 것이지, 원격 저장소를 삭제하는 것이 아님!_



#### (2) 원격 저장소에 업로드하기

> 정확히 하자면, 파일을 업로드하는 것이 아니라 '커밋(버전)'을 업로드 하는 것.
>
> 따라서 로컬 저장소에서 먼저 커밋을 생성해야 업로드 가능. 



1. 로컬 저장소에서 커밋 생성하기 (`git add .` > `git commit -m 'Upload'` )
2. __`git push`__

- 로컬 저장소의 커밋을 원격 저장소에 업로드하는 명령어
- 사용법: `git push <저장소 이름> <브랜치 이름>` 형식으로 작성
  - eg. `git push origin master`
- `-u` 옵션을 사용하면, 다음번 커밋부터는 `<저장소 이름> 과 <브랜치 이름>`을 생략할 수 있다.
